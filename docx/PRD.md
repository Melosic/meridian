\# Meridian 产品需求文档（PRD）

---

\## 1. 产品概述

\### 1.1 产品背景

在闲鱼、转转、eBay等二手交易平台进行买卖的用户越来越多，但多数卖家仅凭记忆或简单笔记记录交易盈亏，缺乏系统化的工具。现有记账软件往往不贴合二手交易场景（如需要记录邮费、商品图片、计算单件利润）。本产品旨在为个人卖家提供一个轻量、直观的利润记录工具，帮助用户清晰掌握每笔交易的收益情况，并支持多账号经营场景下的内部资金结算。

\### 1.2 产品定位

一款面向个人二手卖家的\*\*轻量级利润记账Web应用\*\*，支持记录商品售价、成本、邮费，自动计算利润，支持商品图片、分类管理，并创新性地提供多账号内部成本结算功能（含邮费独立核算）。所有数据支持本地持久化保存，并可选择云端同步（未来版本），部署于Vercel平台，实现快速访问与持续迭代。


\### 1.3 目标用户

\- 闲鱼、转转、eBay等二手平台的个人卖家（特别是多账号经营者）

\- 两人或多人合伙经营二手交易的团队

\- 希望精细化管理各账号盈亏的记账爱好者

\## 3. 界面设计要求

\### 3.1 设计原则

\- \*\*简洁清晰\*\*：去除冗余装饰，突出商品信息和利润数据。

\- \*\*移动优先\*\*：采用响应式设计，保证手机端操作流畅，底部导航栏便于单手操作。

\- \*\*即时反馈\*\*：用户操作后应有明确提示（如Toast通知）。

\- \*\*容错性\*\*：输入框进行必要校验，避免非法数据导致计算错误。

\### 3.2 主要页面布局

\#### 3.2.1 首页（分类视图）

\- \*\*顶部\*\*：标题“Meridian”，视图切换按钮（列表/方格）。

\- \*\*主体\*\*：

&nbsp; - 列表模式：垂直排列分类名称及商品数量，点击进入该分类商品列表。

&nbsp; - 方格模式：网格展示分类卡片，每个卡片显示分类名称、商品数量、代表性图标。

\- \*\*底部导航栏\*\*（移动端）：首页、商品列表、账号管理、结算汇总、设置。

\#### 3.2.2 商品列表页

\- \*\*顶部\*\*：返回按钮、当前分类名称、搜索框、筛选按钮、新增商品按钮。

\- \*\*主体\*\*：商品列表（卡片或列表形式），支持分页加载。每个商品项显示：商品名称、售价、利润（颜色区分）、缩略图、分类标识、结算状态（已结算/未结算）。点击商品进入详情页。

\- \*\*底部\*\*（可选）：批量操作栏（多选后出现，支持批量删除、批量改分类）。

\#### 3.2.3 新增/编辑商品页

\- \*\*表单字段\*\*：

&nbsp; - 商品名称（文本框，必填）

&nbsp; - 分类（下拉选择，可快速跳转添加分类）

&nbsp; - 购买账号（成本支付方，下拉选择，必填）

&nbsp; - 邮费支付账号（下拉选择，必填，默认与购买账号相同）

&nbsp; - 销售账号（下拉选择，必填）

&nbsp; - 售价（数字输入，必填，≥0）

&nbsp; - 成本价（数字输入，必填，≥0）

&nbsp; - 邮费（数字输入，必填，≥0）

&nbsp; - 利润（只读显示，自动计算）

&nbsp; - 图片上传区域（最多5张，支持预览、删除、下载）

\- \*\*动态提示\*\*：根据购买账号、邮费支付账号、销售账号的组合，实时计算并显示内部转账说明，例如：“需从\[销售账号]向\[购买账号]转账\[成本价]元，向\[邮费支付账号]转账\[邮费]元”。

\- \*\*按钮\*\*：保存、取消。

\#### 3.2.4 商品详情页

\- 展示商品所有信息：大图轮播（支持下载）、分类、账号信息（购买、邮费支付、销售）、成本、售价、邮费、利润、结算状态（已结算/未结算）。

\- 若未结算，显示内部转账说明。

\- 若已结算，显示结算后各账号净利润（例如：“A账号净利润：0元，B账号净利润：34元”）。

\- 操作按钮：编辑、删除、标记结算（若未结算）。

\#### 3.2.5 账号管理页

- 列表展示所有账号，支持新增（输入名称）、编辑（重命名）、删除（需确认）。删除前检查关联商品，若有则提供两种处理方式：1）将关联商品的对应账号字段置为"未知"；2）不允许删除（默认）。

\#### 3.2.6 分类管理页

- 类似账号管理，支持增删改分类，删除时检查商品关联，提供两种处理方式：1）将关联商品的分类字段置为"无分类"；2）不允许删除（默认）。

\#### 3.2.7 内部结算汇总页

\- \*\*顶部\*\*：概览卡片，按账号展示应收总额（作为销售账号应转出的总成本+邮费）、应付总额（作为成本/邮费支付方应收到的补偿）、净额。

\- \*\*待结算列表\*\*：列出所有未标记结算的商品，每行包含商品名、购买账号、邮费支付账号、销售账号、成本价、邮费、应转账明细（分账号），操作列有“标记结算”按钮。支持多选批量标记，以及“一键结算所有”按钮。

\- \*\*已结算列表\*\*（可折叠）：显示已标记结算的记录，同样支持导出。

\- \*\*导出按钮\*\*：导出当前待结算清单或全部记录为Excel/CSV。

\#### 3.2.8 导入预览页

\- 用户选择备份文件后，展示预览表格（前10条数据），让用户确认字段映射和内容，确认后执行导入（可选择覆盖或合并）。

#### 3.2.9 设置页

- **数据管理**：清除本地数据（需二次确认）、导出全部数据为JSON/CSV、查看当前存储使用量（商品数、图片数、预估容量）

- **关于**：显示当前版本号、开发者联系方式

\### 3.3 交互细节

\- 利润为负时，数字显示为红色。

\- 上传图片支持拖拽上传，单张图片大小限制≤2MB，格式jpg/png/gif/webp。

\- 删除操作需弹出确认框，批量删除时提示“确认删除选中的X件商品？”。

\- 内部结算提示在商品表单、详情页、结算汇总页显著显示。

\- 标记已结算后，对应记录从待结算列表移至已结算，并更新概览数据。

\- 空状态：当账号/分类/商品列表为空时，显示插图和文字提示，并提供“立即创建”按钮。

\- 移动端Toast提示位于屏幕底部导航栏上方，持续2秒自动消失。

\## 4. 技术栈建议（Vercel部署）

为充分利用Vercel平台能力，实现快速部署与平滑扩展，推荐以下技术栈：

\### 4.1 前端

\- \*\*框架\*\*：Next.js 14 (App Router)

\- \*\*UI组件库\*\*：Ant Design Mobile 或 Material-UI（响应式）

\- \*\*状态管理\*\*：Zustand（轻量，适合管理账号、分类、结算状态等全局数据）

\- \*\*样式方案\*\*：Tailwind CSS + CSS Modules

\- \*\*数据获取与缓存\*\*：SWR（用于未来云端API）

\### 4.2 后端（云端同步版本）

\- \*\*方案A：Next.js API Routes\*\* + \*\*Supabase\*\*（推荐）

&nbsp; - Supabase提供PostgreSQL数据库、认证、存储，API Routes作为BFF层处理复杂逻辑。

\- \*\*方案B：Vercel Postgres + Vercel Blob\*\* + \*\*NextAuth.js\*\*（完全Vercel生态）

\### 4.3 数据库

\- \*\*本地存储（MVP）\*\*：IndexedDB（通过`localForage`或`idb`封装）

&nbsp; - 数据结构需支持商品、账号、分类、结算状态等实体及其关联。

\- \*\*云端存储（迭代）\*\*：Supabase PostgreSQL（关系型数据库，便于多表关联查询）

\### 4.4 图片存储

\- \*\*本地存储（MVP）\*\*：图片转为Base64或Blob存入IndexedDB（注意容量限制：浏览器配额通常为50-100MB，建议单图≤500KB，总图片数建议≤100张，超出提示用户清理或考虑云端存储）

\- \*\*云端存储（迭代）\*\*：Supabase Storage 或 Vercel Blob

\### 4.5 部署与运维

\- \*\*代码托管\*\*：GitHub

\- \*\*部署平台\*\*：Vercel（自动部署，支持环境变量、预览分支）

\- \*\*环境变量\*\*：在Vercel项目设置中配置Supabase URL、Anon Key等

\### 4.6 数据模型扩展

为支持新增功能，商品记录需包含以下字段：

```json

{

&nbsp; "id": "uuid",

&nbsp; "name": "商品名称",

&nbsp; "categoryId": "分类ID",

&nbsp; "buyAccountId": "购买账号ID",       // 成本支付账号

&nbsp; "shippingAccountId": "邮费支付账号ID", // 新增字段

&nbsp; "sellAccountId": "销售账号ID",

&nbsp; "price": 1200,

&nbsp; "cost": 800,

&nbsp; "shipping": 12,

&nbsp; "profit": <!-- 388, --> 利润为计算字段，不持久化存储

&nbsp; "images": \["base64string1", ...],  // 最多5张

&nbsp; "settled": false,                  // 是否已内部结算
 
&nbsp; "createdAt": "2025-07-10T10:00:00Z",

&nbsp; "updatedAt": "2025-07-10T10:00:00Z",

&nbsp; "version": 1  // 数据模型版本号，用于迁移

}

```

账号表、分类表为独立存储。

\### 4.7 技术选型总结（MVP + 可扩展）

| 层次       | MVP方案                          | 可扩展方案（未来）                |

|------------|----------------------------------|-----------------------------------|

| 前端框架   | Next.js (静态导出)               | Next.js + API Routes              |

| 数据库     | IndexedDB (localForage)          | Supabase PostgreSQL               |

| 图片存储   | IndexedDB (Base64/Blob)          | Supabase Storage / Vercel Blob    |

| 认证       | 无（纯本地）                     | Supabase Auth / NextAuth.js       |

| 部署       | Vercel（静态托管）               | Vercel（全栈Serverless）          |

\## 5. 非功能性需求

\### 5.1 性能

\- 页面首次加载时间 ≤ 2秒（生产环境优化后）。

\- 商品列表滚动流畅，图片懒加载，分页每页20条（可根据实际性能测试调整）。

\- 图片上传前压缩（限制大小≤2MB，支持格式jpg/png/gif/webp）。

\### 5.2 安全

\- \*\*本地版本\*\*：对用户输入进行转义，防止XSS；图片格式限制。

\- \*\*云端版本\*\*：HTTPS传输；用户认证与权限隔离；数据库RLS策略确保数据隔离。

\### 5.3 可用性

\- 支持主流浏览器最新版本。

\- 移动端适配良好，按钮大小适合手指点击（≥44x44px），底部导航栏高度适中。

\- 关键操作（删除、标记结算）有二次确认。

\### 5.4 可维护性

\- 代码遵循ESLint规范，使用Prettier格式化。

\- 组件拆分合理，逻辑复用采用自定义Hooks。

\- 数据存储层抽象，便于未来替换为云端API。

\### 5.5 扩展性

\- 数据结构预留字段：标签、交易平台、成交日期等。

\- 图片存储抽象，可无缝切换存储后端。

\- 数据版本管理：在数据模型中添加version字段，数据结构变更时自动执行迁移逻辑，确保向后兼容。

\### 7.2 结算后净利润计算示例

\- 商品：成本50元（A账号支付），邮费6元（A账号支付），售价90元（B账号销售）。

\- 内部转账：B需向A转账56元。

\- 结算后：

&nbsp; - A账号净利润 = 收到56 - 支出56 = 0元

&nbsp; - B账号净利润 = 90 - 56 = 34元

\- 显示格式：“A账号净利润：0元，B账号净利润：34元”
